For years, engineering culture treated waterfall as a cautionary tale. Plans were expensive, requirements changed, and teams needed to ship visible increments to learn what mattered.

That critique was mostly right for human-only delivery.

Agentic delivery changes the economics.

When code is generated by parallel agents, architecture can drift faster than ever. Each agent is strong at local optimization: complete the task, satisfy the prompt, move on. Without a global plan, clean local changes can still produce a fragmented system.

In this environment, waterfall is not nostalgia. It is control.

## Waterfall for agentic systems

The classic objection to waterfall is planning cost. Exhaustive up-front planning took too long and delayed feedback.

With agents, planning artifacts can be produced quickly: formal models, interface contracts, state transition maps, threat models, test matrices, and decision logs. The bottleneck is no longer writing plans. The bottleneck is keeping generated code aligned to architectural intent.

So the sequence should be explicit:

1. Exhaustively model and plan the system.
2. Review and approve the plan as the source of truth.
3. Let agents implement within that approved frame.

Iteration still exists, but the loop changes. We do not iterate from random code into architecture. We iterate from architecture into code.

## Skills-Driven Development

This is the pattern I call **Skills-Driven Development (SDD)**.

In SDD, the skill is committed before the code.

A skill is not just a prompt template. It is an executable reasoning artifact that contains:

- domain constraints,
- invariants,
- allowed state transitions,
- architectural boundaries,
- and proof obligations for high-risk logic.

The workflow is deliberate:

1. Write the skill artifact.
2. Review the skill for logic errors, security gaps, and architectural inconsistencies.
3. Commit the skill as a versioned baseline.
4. Allow agents to apply that skill to generate implementation code.
5. Verify generated code against the committed skill.

This reverses the usual review burden. Instead of trying to infer intent from a large code diff, reviewers validate intent first and execution second.

## Authorization as the reference use case

Authorization is where SDD has immediate value.

Most authorization systems drift through exceptions: one endpoint-level role check here, one tenant carveout there, one support bypass added during an incident. Over time, policy becomes distributed folklore.

An SDD workflow starts from a formal authorization skill:

- define principals, resources, actions, and context,
- state invariants like "no cross-tenant read" and "elevation must be time-bounded and attributable",
- specify decision paths for allow, deny, and escalation,
- and encode required evidence for every privileged action.

Agents then implement policy checks from this baseline instead of inventing behavior endpoint by endpoint.

You still audit, but the audit changes shape. Instead of reconstructing policy intent from production code, you verify conformance to a committed logic model.

## Practical adoption path

Teams do not need a full process rewrite to start.

1. Pick one high-risk domain (authorization is ideal).
2. Create a first skill artifact with explicit invariants and failure modes.
3. Add review gates that require skill approval before implementation PRs.
4. Require traceability from implementation changes back to skill clauses.
5. Track drift as a first-class metric: where code behavior diverges from the committed skill.

The key principle is simple: in agentic engineering, architecture must be centralized before coding is parallelized.

Waterfall did not become correct for every software problem. But for agentic systems, plan-first development is no longer a costly luxury. It is the fastest path to coherence, security, and predictable change.
